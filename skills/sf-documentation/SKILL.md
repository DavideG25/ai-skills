  Input can be an object name (e.g. Quote__c) or a process description (e.g. "order creation").

  "documento di processo", "process document", "cos'è questo oggetto", "spiega l'oggetto",
  or when an object name or process name is passed for documentation purposes.

# SF Documentation — Orchestrator

You run in a **forked context** — isolated from the user's conversation.
The user invoked you with: **$ARGUMENTS** (object name or process description).

## Step 0: Load existing context

1. If `.claude/context/_index.md` exists → scan it for $ARGUMENTS. If a matching entry exists, read the corresponding `.claude/context/[Name].md` — use it to enrich the document without re-scanning everything.
2. If `.claude/rules/metadata-rules.md` exists → read it. Apply project-specific field and naming conventions.

## Step 1: Detect input type

Determine what $ARGUMENTS refers to:

**Object** — ends in `__c`, or matches a standard object name (Account, Contact, Opportunity, Lead, Case, Quote, Order...), or the user explicitly said "object" / "oggetto".

**Process** — a verb phrase or event description (e.g. "order creation", "quote approval", "account insert"), or the user said "process" / "processo".

If ambiguous, ask the user: "Stai chiedendo la documentazione dell'oggetto [X] o del processo [X]?"

## Step 2: Scan — adapt to input type

### If OBJECT:

**If Task tool is available** — spawn in parallel using Task():
1. **sf-object-scanner** — field list, relationships, picklist values, record types
2. **sf-trigger-scanner** — triggers on this object
3. **sf-flow-scanner** — record-triggered flows on this object
4. **sf-validation-scanner** — validation rules on this object

**If Task tool is NOT available** — run sequentially:
1. **Object metadata**: `Glob "*[ObjectName]*.object-meta.xml"`. Read it: extract fields (label, type, required), relationships, record types, picklist values.
2. **Triggers**: `Glob "*.trigger"`. Read each matching the object — note when it fires and what it does in one line.
3. **Flows**: `Glob "*.flow-meta.xml"`. For each, check `<object>` tag. Note type, trigger condition, main actions.
4. **Validation rules**: `Glob "*[ObjectName]*.validationRule-meta.xml"` or read from the object XML. Note condition and error message in business language.

### If PROCESS:

**If Task tool is available** — spawn in parallel using Task():
1. **sf-trigger-scanner** — Object + event
2. **sf-flow-scanner** — Object + event
3. **sf-validation-scanner** — Object + event
4. **sf-integration-scanner** — Object + event
5. **sf-workflow-scanner** — Object + event

**If Task tool is NOT available** — run the same sequential scan from sf-process-discovery Step 2 fallback path, focused on the object and event extracted from $ARGUMENTS.

Wait for all agents to return.

## Step 3: Build the document

Compose a Markdown document with both sections. Write the functional section in plain language — no field API names, no technical jargon. Write the technical section for developers.

```markdown
# [ObjectName / Process Name] — Documentation

> Generated on [date] by sf-documentation

---

## Functional Overview

### What it is / What it does
[3-5 sentences in plain business language. Who uses it, when, and why.]

### Key fields
| Field | Purpose |
|-------|---------|
| [Label] | [what it represents in business terms] |
| ... | ... |

[Only fields that matter to the business — skip system fields and technical internals]

### Business rules
[Validation rules and constraints in plain language — no formulas]
- [Rule: when X condition, Y is blocked/required because Z]

### How it fits in the process
[If object: which processes create/update/use it]
[If process: the sequence of steps from trigger to outcome, in business language]

---

## Technical Reference

### Object / Trigger
[Object API name, main fields (API name + type), record types if any]

### Automations (execution order)
1. [Automation name] — [type: Trigger/Flow/Validation/Workflow] — [fires when + what it does]
2. ...

### Relationships
| Relationship | Type | Object |
|---|---|---|
| [Field API name] | Lookup/Master-Detail | [Related object] |

### Integrations
[Callouts, platform events, external systems involved — or "None"]

### Key dependencies
[Other objects or classes that interact with this one]

---

*Generated by sf-documentation — sf-ai-skills*
```

## Step 4: Save the document

Create `docs/` if it doesn't exist.
Save as `docs/[Name].md`.

Then update `.claude/context/`:
- Write a compact version to `.claude/context/[Name].md` (purpose, key fields, automations list)
- Update `.claude/context/_index.md` with a one-line entry

## Step 5: Generate .docx (optional)

```bash
pandoc --version 2>/dev/null | head -1
```

If available: `pandoc docs/[Name].md -o docs/[Name].docx`
If not: mention it in the final message.

## Step 6: Deliver

Return to the user:
- Brief summary of what was documented (2-3 sentences)
- Paths of files saved
- Any notable automation or dependency worth highlighting
